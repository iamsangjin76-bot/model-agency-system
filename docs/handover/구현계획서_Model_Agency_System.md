## Phase E — 데이터 기반 구축

> **목표**: 앱을 열면 실제 데이터가 보이는 상태 만들기

지금 시스템의 가장 큰 문제는 **DB가 텅 비어있다**는 겁니다. 차트도 있고 화면도 있는데 데이터가 없어서 전부 "데이터 없음"입니다. 이걸 먼저 해결해야 다음 작업의 결과를 눈으로 확인할 수 있습니다.

### E-1. DB 스키마 점검 + 시드 데이터 생성

**난이도**: 🔴 Complex (DB 스키마 확인 + 다중 테이블 데이터 삽입)

**할 일**: 현재 SQLite DB의 테이블 구조를 정확히 파악하고, 한국 모델 에이전시 맥락에 맞는 샘플 데이터를 넣습니다.

**시드 데이터 예상 범위**:
- 모델 10~15명 (이름, 성별, 키, 몸무게, 나이, 소속 등)
- 캐스팅 공고 5~8건 (브랜드명, 촬영일, 장소, 요구 조건 등)
- 정산 레코드 10~20건 (모델별 수입, 에이전시 수수료, 지급일 등)
- 사용자 계정 2~3개 (admin + 매니저)

**완료 기준**: 로그인 → 대시보드에서 차트 3개에 실제 수치가 표시됨

**예상 세션**: 1~2세션 (스키마 파악 1 + 시드 스크립트 작성·실행 1)

**ADK 흐름**: architect(스키마 분석) → executor(시드 스크립트) → reviewer(데이터 정합성)

### E-2. 대시보드 "최근 활동" 동적 연결

**난이도**: 🟡 Medium (기존 UI + API 연결)

**할 일**: 하드코딩된 "김모델 / 홍길동 / 박지민 / 이수진" 더미 데이터를 실제 DB에서 가져오는 API로 교체합니다.

**완료 기준**: 대시보드 "최근 활동" 패널이 실제 최근 모델 등록·캐스팅·정산 이벤트를 시간순으로 표시

**예상 세션**: 1세션

**ADK 흐름**: planner → executor → reviewer

### E-3. 대시보드 차트 실데이터 연결 확인

**난이도**: 🟢 Simple ~ 🟡 Medium

**할 일**: E-1에서 시드 데이터를 넣은 뒤, 차트 3개가 정확한 수치를 보여주는지 확인합니다. API가 이미 있다면 Simple(확인만), API 수정이 필요하면 Medium.

**완료 기준**: 성별 분포 PieChart + 캐스팅 현황 BarChart + 정산 요약 BarChart 모두 시드 데이터 반영

**예상 세션**: E-1과 같은 세션 또는 별도 1세션

## Phase F — 핵심 도메인 기능 완성

> **목표**: 모델·캐스팅·정산 3대 도메인의 CRUD를 실사용 수준으로 올리기

현재 엔드포인트 9개가 있으나, 실제로 등록·수정·삭제·조회가 프론트엔드 UI와 완전히 연결되어 사용자가 마우스 클릭만으로 모든 작업을 할 수 있는지는 확인이 필요합니다.

### F-1. 모델 관리 CRUD 완성

**난이도**: 🔴 Complex (새 SPEC 가능성 — SPEC-MODEL-001)

**할 일**:
- 모델 목록 조회 (검색·필터·정렬)
- 모델 상세 조회 (프로필 카드)
- 모델 신규 등록 (폼 → API → DB)
- 모델 정보 수정
- 모델 삭제 (소프트 삭제 권장)
- 모델 프로필 사진 표시 (Phase G의 파일 업로드와 연계)

**완료 기준**: UI에서 모델 등록 → 목록에 표시 → 클릭해서 상세 확인 → 수정 → 삭제 전체 흐름이 동작

**예상 세션**: 2~4세션

**ADK 흐름**: architect(SPEC 설계) → planner → executor → reviewer → security-auditor(입력 검증)

### F-2. 캐스팅 관리 CRUD 완성

**난이도**: 🔴 Complex (새 SPEC — SPEC-CASTING-001)

**할 일**:
- 캐스팅 공고 등록 (브랜드, 날짜, 장소, 요구 조건, 보수)
- 캐스팅 목록 조회 (상태별 필터: 모집중/진행중/완료)
- 캐스팅에 모델 배정 (모델↔캐스팅 N:N 관계)
- 캐스팅 상태 변경 (모집중 → 진행중 → 완료)
- 캐스팅 수정·삭제

**완료 기준**: 캐스팅 공고 생성 → 모델 배정 → 상태 변경 → 완료 처리까지 전체 워크플로 동작

**예상 세션**: 3~5세션

**ADK 흐름**: architect(SPEC + DB 스키마) → planner → executor → reviewer → security-auditor

### F-3. 정산 관리 CRUD 완성

**난이도**: 🔴 Complex (새 SPEC — SPEC-SETTLEMENT-001)

**할 일**:
- 정산 레코드 생성 (캐스팅 완료 → 모델 보수 + 에이전시 수수료 계산)
- 정산 목록 조회 (기간별, 모델별, 상태별)
- 정산 상태 관리 (미지급 → 지급완료)
- 정산 수정·삭제
- 정산 요약 통계 (월별 수입/지출)

**완료 기준**: 캐스팅 완료 → 정산 생성 → 지급 처리까지 금액 흐름이 완전히 추적됨

**예상 세션**: 2~4세션

**ADK 흐름**: architect(SPEC + 금액 계산 로직) → planner → executor → reviewer → security-auditor

## Phase G — 부가 기능

> **목표**: 실사용 편의성을 높이는 기능 추가

### G-1. 알림 시스템

**난이도**: 🔴 Complex (새 SPEC — SPEC-NOTIF-001)

**할 일**:
- 백엔드: 알림 서비스 + 알림 테이블
- 프론트엔드: 대시보드 상단 벨 아이콘 + 알림 드롭다운
- 알림 종류: 새 캐스팅 등록, 캐스팅 상태 변경, 정산 완료, 계약 만료 임박 등
- 읽음/안읽음 처리

**완료 기준**: 캐스팅 상태 변경 시 관련 알림이 자동 생성되고, 벨 아이콘에 미읽음 개수 표시

**예상 세션**: 2~3세션

### G-2. 파일 업로드 검증

**난이도**: 🟡 Medium ~ 🔴 Complex

**할 일**:
- 모델 프로필 사진 업로드 (JPEG/PNG, 크기 제한)
- 포트폴리오 이미지 다중 업로드
- 파일 타입·크기 서버측 검증
- 이미지 미리보기 (프론트엔드)
- 업로드 파일 저장 경로 관리

**완료 기준**: 모델 등록/수정 시 사진 업로드 → 프로필에 표시 → 삭제 가능

**예상 세션**: 2~3세션

### G-3. 검색·필터 고도화

**난이도**: 🟡 Medium

**할 일**:
- 모델 검색: 이름, 성별, 키 범위, 나이 범위
- 캐스팅 검색: 브랜드, 날짜 범위, 상태
- 정산 검색: 기간, 모델명, 상태
- 정렬 옵션 (이름순, 날짜순, 금액순)

**완료 기준**: 각 목록 페이지에서 복합 조건 검색이 동작

**예상 세션**: 1~2세션

## Phase H — UX 완성

> **목표**: 사용자 경험을 전문 소프트웨어 수준으로 끌어올리기

### H-1. 반응형 레이아웃

**난이도**: 🟡 Medium

**할 일**:
- 사이드바 접기/펼치기 (모바일에서는 오버레이)
- 테이블 → 카드 뷰 전환 (좁은 화면)
- 폼 레이아웃 조정
- 최소 지원 해상도 정의 (1024px 이상 권장)

**완료 기준**: 1024px ~ 1920px 범위에서 깨지지 않고 사용 가능

**예상 세션**: 2~3세션

### H-2. 다크모드

**난이도**: 🟡 Medium

**할 일**:
- CSS 변수 기반 테마 시스템
- 토글 버튼 (헤더 우측)
- 시스템 설정 연동 (prefers-color-scheme)
- 차트 색상 다크모드 대응

**완료 기준**: 토글 전환 시 전체 화면이 다크/라이트로 즉시 변경, 차트 포함

**예상 세션**: 1~2세션

### H-3. 에러 처리·로딩 UX

**난이도**: 🟡 Medium

**할 일**:
- API 에러 시 사용자 친화적 메시지 (토스트 알림)
- 네트워크 오류 대응
- 로딩 스피너·스켈레톤 UI
- 404 페이지
- 폼 유효성 검증 메시지 (프론트엔드)

**완료 기준**: 어떤 에러 상황에서도 백지 화면이 뜨지 않음

**예상 세션**: 1~2세션

## Phase I — 배포 준비

> **목표**: 실제로 다른 PC에 설치해서 쓸 수 있는 상태 만들기

### I-1. Electron 패키징

**난이도**: 🔴 Complex

**할 일**:
- Electron 빌드 설정 (electron-builder)
- 백엔드를 Electron 프로세스 내에서 자동 시작
- Windows 설치 파일(.exe) 생성
- 자동 업데이트 메커니즘 (선택)

**완료 기준**: 빌드된 .exe 파일을 더블클릭하면 앱이 실행되고, 별도 서버 실행 없이 로그인~대시보드까지 동작

**예상 세션**: 3~5세션

### I-2. 백업·복원

**난이도**: 🟡 Medium

**할 일**:
- SQLite DB 파일 백업 (날짜별 자동)
- 백업 복원 기능 (설정 메뉴)
- 업로드된 파일 백업 포함

**완료 기준**: 설정 → 백업 생성 → 새 환경에서 복원 → 모든 데이터 유지

**예상 세션**: 1~2세션

### I-3. 테스트 기반 구축

**난이도**: 🟡 Medium ~ 🔴 Complex

**할 일**:
- 백엔드 핵심 API 테스트 (pytest)
- 프론트엔드 컴포넌트 테스트 (vitest)
- E2E 테스트 핵심 시나리오 (로그인 → 모델 등록 → 캐스팅 배정)

**완료 기준**: `pytest` + `npm test` 통과, 핵심 워크플로 E2E 3개 이상

**예상 세션**: 2~4세션

## 🎯 권장 진행 순서

### 즉시 시작 권장 (가치 대비 노력 최고)

**Phase E-1 (시드 데이터)** → 이게 없으면 모든 화면이 빈 상태입니다. 가장 먼저.

### 그 다음

**Phase E-2 + E-3** → 대시보드가 살아나면 진행 상황이 눈에 보이기 시작합니다.

### 핵심 개발

**Phase F (도메인 CRUD)** → 이 시스템의 존재 이유입니다. E 이후 가장 중요.

### 선택적 / 상황에 따라

- Phase G, H: 기본 기능이 동작한 뒤 사용 편의성 개선
- Phase I: 다른 PC에 배포하거나 팀원에게 공유할 때
- Phase D-3 (git filter-repo): 우선순위 최하. clone 속도 체감 문제 있을 때만

3. **Phase G~H 중 급한 것**: 알림이 급한지, 반응형이 급한지, 다른 기능이 더 필요한지
4. **Phase I 시점**: Electron 패키징은 언제쯤 필요한지 (당장? 기능 완성 후?)
5. **추가 기능**: 이 계획에 없지만 필요한 기능이 있는지 (예: 계약서 관리, 스케줄 캘린더, 엑셀 내보내기 등)
