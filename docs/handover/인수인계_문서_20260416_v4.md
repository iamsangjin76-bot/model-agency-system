## 🎓 이번 세션에서 배운 교훈

1. **비개발자의 메타 판단은 과소평가하지 말 것**
   - 사용자가 두 번 반박함 (결정 3 의 "아키텍트가 채우는 구조" 지적 + 결정 4 의 4줄 복원)
   - 두 번 다 아키텍트 논리의 약점을 정확히 찾음
   - 교훈: "비개발자 = 코드 모름" 이지만 "비개발자 = 프로세스 판단 못함" 은 아님. 오히려 코드 제약이 없어 상위 레벨 판단에 유리할 수 있음.

2. **"누가 템플릿을 채우는가" 가 템플릿 설계를 결정**
   - 사용자 타이핑 부담 vs 아키텍트가 채우는 워크플로는 다른 설계 기준 필요
   - 전자 → 최소화·단순화 우선
   - 후자 → 정보 밀도·표준 일치 우선
   - 이번 프로젝트는 후자 → 4줄 유지가 맞음

3. **Path B (단계 분할) 가 안전장치로 잘 작동**
   - 한 세션에 몰아붙였으면 답변 끊기거나 뒷부분 부실해졌을 것
   - 설계 → 실행 분할 = 각 단계에 충분한 공간 확보

4. **5대 결정 질문이 모두 "실행 전 설계" 층위였던 것이 좋았음**
   - 기능 스펙은 실행하면서 알아가도 되지만, 프로세스 설계는 먼저 고정해야 갈림길이 줄어듦
   - 앞으로 새 큰 주제 다룰 때도 이 "결정 포인트 N개부터 확정" 패턴 재사용 가치 있음
