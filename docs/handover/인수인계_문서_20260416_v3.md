### `autopus.yaml` 핵심 설정

- **Mode**: `full` (모든 기능 활성)
- **Platforms**: `claude-code`, `codex`, `gemini-cli`, `opencode` (4-플랫폼)
- **Model routing**: economy/standard = Sonnet 4.6, premium = Opus 4.6
- **Quality presets**: `balanced` (기본, Opus+Sonnet 혼합) / `ultra` (전부 Opus)
- **Methodology**: TDD 강제, review gate 활성
- **Lore commits**: 활성, 필수 trailer = `Constraint`
- **Orchestra**: brainstorm/plan/review/secure 명령에 debate/consensus 사용 (현재 미활용)
- **SPEC format**: `SPEC-{DOMAIN}-{NUMBER}`

### 활성 Git 훅 2개

| 훅 | 동작 |
|---|---|
| `pre-commit` | `auto check --arch --staged` — 스테이징된 파일의 아키텍처 룰 검증 |
| `commit-msg` | `auto check --lore + auto lore validate` — Lore 포맷 · `Constraint` trailer · 🐙 sign-off 강제 |

> `post-commit`, `pre-push` 등 다른 커스텀 훅은 없음 (`.sample` 파일만 남음).
> 별도로 `.claude/hooks/autopus/` 에 7개 하네스 내부 훅이 있음 (session stop, CI failure reaction 등).

### 보고된 기타 워크플로 규칙

- **Supervisor Contract**: 메인 세션은 단순 라우터가 아닌 phase/gate 감독자 역할
- **Worker Contracts**: 모든 서브에이전트 프롬프트에 owned files, forbidden scope, completion criteria, return format 명시
- **Review Convergence**: discovery ≠ verification 분리, 무한 재스캔 금지
- **Objective Reasoning rule**: 증거 기반 응답, trade-off 공개, calibrated confidence, 공허한 동의/반대 금지

## 🩺 Phase B — 아키텍트 진단

### Keep / Simplify / Remove / Defer 매트릭스

#### ✅ KEEP (지금 당장 가치)
- **Git 훅 2개** (arch check + Lore) — 오늘 실전 증명
- **주력 에이전트 5개**: `explorer`, `executor`, `debugger`, `reviewer`, `architect`
- **Supervisor/Worker Contract 원칙** — 비개발자에게 오히려 안전장치

#### 🔧 SIMPLIFY (유지하되 진입장벽 낮추기)
- **Worker Contract** → 비개발자용 **"3줄 템플릿"** 으로 축약
- **40+ skills** → 자주 쓸 5~7개만 "주력 스킬 리스트" 로 추림
- **16 agents 소개** → "언제 어느 에이전트 부를지" 1페이지 치트시트

#### 🗑️ REMOVE (실질 해로움 또는 버그)
- **`constraints.yaml` 의 Go deny 패턴 5개** — 설정 버그. Python/FastAPI + React/TS 프로젝트에 Go 룰 박혀있음. 다음 작업 후보 1순위.

#### ⏸️ DEFER (팀·SPEC 규모 커지면 재검토)
- **Orchestra multi-provider debate** — 1 SPEC 단계엔 과함
- **4-platform mirroring** — claude-code 단일 사용 중엔 동기화 면제
- **review gate debate strategy** — SPEC 3개+ 쌓이면 활성

### 핵심 인사이트 2가지

**1. "비개발자 솔로 플레이" 를 1급 시민으로 선언**
현재 ADK 는 암묵적으로 "팀 + 다중 AI 공급자 + 다수 SPEC" 전제. 실상은 1명의 비개발자 + 1개의 AI 하네스. 워크플로 문서가 이 현실을 **명시적으로** 받아들여야 함.

**2. "오케스트레이션 비용 < 실제 복잡도" 가 기본 철학**
모든 작업에 debate 걸면 결정 속도 느려져 비개발자에겐 독. **"단순 작업엔 단순 워크플로, 복잡 작업엔 ADK 풀파워"**.
오늘 #9 작업이 좋은 반례 — 한 줄 추가에 debate 걸었으면 몇 시간 낭비.

## 🗺️ Phase C 설계 브리프 (새 창의 미션)

### 목표
`.autopus/WORKFLOW.md` 한 파일로 **"이 프로젝트의 표준 워크플로"** 정의.
비개발자가 매 작업마다 뭘 어느 에이전트에게 어떻게 지시할지 **1페이지**에서 해결되게.

### 문서에 들어갈 섹션 (제안 초안)

1. **작업 난이도 분류** — Simple / Medium / Complex
   - Simple 예: 의존성 추가, 오타 수정, 주석 보완 → 설계 대화 생략, 바로 executor
   - Medium 예: 기능 추가(단일 SPEC 내), 버그 수정, 리팩토링 → planner → executor → reviewer
   - Complex 예: 새 SPEC, 아키텍처 변경, 성능 개선 → architect → planner → executor → reviewer → security-auditor
2. **단계별 표준 지시문 템플릿** — 복붙 가능한 3종 세트
3. **에이전트 치트시트** — 언제 어느 에이전트를 부를지 결정 트리
4. **비개발자 체크리스트** — 작업 전/중/후 확인 포인트
5. **Lore 커밋 템플릿** — `type(scope)` 별 샘플 3~5개

### Phase C 시작 시 결정해야 할 포인트
1. 이 문서를 `.autopus/WORKFLOW.md` 에 **신규** vs `AGENTS.md` 에 **증보**?
   → 아키텍트 잠정 의견: **신규 파일**. AGENTS.md 는 "에이전트 정의" 라 범위 다름. 그리고 `.autopus/` 아래는 프로젝트 전용 규약 보관소.
2. 에이전트 치트시트 깊이 — 주력 5개만 vs 전체 16개 (권장: **5개 집중 + 나머지 1줄 요약**)
3. 작업 난이도 판정 기준을 어떻게 객관화할 것인가 (파일 수? 예상 시간? 영향 범위?)
4. Worker Contract 3줄 템플릿의 구체 항목 (예: `목표 / 범위 / 금지` vs `목표 / 완료기준 / 반환형식`)
5. `constraints.yaml` Go 패턴 제거를 같은 문서에서 다룰지 별도 커밋으로 분리할지

### Phase C 예상 산출물
- 파일: `.autopus/WORKFLOW.md` (신규)
- 커밋 1개 (Lore format):
  ```
  docs(workflow): establish non-developer solo-play workflow
  ```
- 관련 커밋 (선택, 별도 분리 권장):
  ```
  chore(context): replace go-specific deny patterns with python/react rules
  ```
